الـ Whitepaper (مُحدّث يشمل الفوركس والفيوتشر)

ملخّص تقني وتشغيلي يشرح آلية العمل، دمج الفيوتشر، وحوكمة التوكن.

1. الملخص التنفيذي

RAM هو محرك لإدارة التعرض والـ Arbitrage يعمل بنمط Dual-Account: حساب A (High-Risk / Aggressive) وحساب B (Reverse / Hedged). يتم توسيع الفكرة لتشمل أسواق Crypto Futures (Perpetuals & Quarterly futures) بحيث نستفيد من ميكانيكيات التسوية، التمويل (Funding), والـ Liquidations.

2. الأسباب والفرص

الفيوتشر في الكريبتو يشترك مع الفوركس في نفس آليات الرافعة والسوق الثنائي الاتجاه (long/short). اختلاف التنفيذ، حجم السيولة، أو ظروف التسعير بين منصات/حسابات يخلق فرص لإدارة المخاطر وتحويل خسائر من حساب إلى أرباح في حساب آخر بطريقة محسوبة وآمنة.

3. الدعم التقني للأسواق

الـ Engine يدعم حالياً:

  • MT4 / MT5 (EA ↔ API/WS)
  • Binance Futures (USDT-M, COIN-M) عبر REST & WebSocket
  • Bybit, OKX, MEXC, Bitget — دعم عبر REST/WebSocket وAPI Key
  • Pairs: Forex majors & minors، Crypto pairs & perpetuals (BTCUSDT, ETHUSDT...)

4. كيف يعمل مع الفيوتشر (Technical)

  1. Signal & Execution: EA/Adapter يرسل التنفيذ (open/close/order-fill) + timestamps + fees إلى Backend.
  2. Reverse Engine: يقرر فتح صفقة معاكسة على حساب Hedged وفق قواعد: leverage mapping, dynamic sizing (USD-equivalent), funding cost adjustment.
  3. Funding & Fees Adjustment: عند فتح مراكز في الفيوتشر، Engine يحسب تكلفة الـ funding ويعدل حجم التداول لتأمين الربحية الصافية.
  4. Liquidation-aware sizing: Engine يطبق قواعد تمنع تعريض حساب Hedged لخطر تصفية مبالغ فيه (max leverage cap).

5. Architecture (محدث للفيوتشر)

[MT5 EA]       [Binance Adapter]    [Bybit Adapter]   ...
   |                 |                    |
   +-----> [API Gateway / Auth / WS Server] <-----+
                     |                          |
                 [Reverse Engine]               |
                     |                          |
               [DB: Postgres / Logs]            |
                     |                          |
                 [Dashboard (React)] --------- Web3 (ethers.js)

6. Risk Controls specific to Futures

  • Mapping of leverage levels between accounts (e.g., A: 50x, B: 3x).
  • Dynamic sizing to keep USD-equivalent exposure aligned across assets.
  • Funding fee offsets: compute expected funding per period and net it from P&L.
  • Max open positions and daily drawdown kill-switch.

7. Data Requirements & Latency

الفيوتشر حساس للتأخير: استخدم WebSocket لfills وأسعار السوق، احفظ توقيت السيرفر (UTC) لكل حدث، وطبّق reconcile لوغاريثم لموازنة اختلاف التوقيت.

8. Architecture: Exchanges Adapters

كل Exchange Adapter يقوم بـ:

  • مصادقة API Key/Secret (محفوظة مشفّرة في Vault)
  • إرسال أوامر (POST) واستقبال fills عبر WebSocket
  • حساب fees و funding و margin requirements

Futures Support — الملخص العملي

ندعم نقل الفكرة بالكامل إلى أسواق التداول المستقبليّة. النقاط الأساسية:

  • Perpetuals vs Quarterly: التعامل مع funding في الـ Perpetuals و settlement في الـ Quarterly.
  • Cross-exchange execution: فتح/إغلاق مراكز على منصات مختلفة مع مراعاة فروق الأسعار والتنفيذ.
  • Adapters: Binance, Bybit, OKX, MEXC, Bitget — كل واحد بموديل fees/margin مختلف.

Integration checklist

  1. API keys per account (read/write) — تخزين مشفّر.
  2. WebSocket for price & fills (low latency).
  3. Reconcile engine to match fills with logs (idempotency).
  4. Per-exchange funding & fee calculation module.

تحذيرات قانونية وأخلاقيات (مهم جدًا)

  • لا تُستخدم هذه الآليات لاستغلال ثغرات أو احتيال: أي استغلال أو خداع قد يعرضك للمساءلة الجنائية والمدنية.
  • قبل جمع أموال، إجراء مراجعة قانونية (Securities Laws): قد تُعتبر بعض أشكال التكييف المالي security/token offering في بعض الدول.
  • لو ربطت النظام بمنصات تداول، احصل على موافقات أو التزم بشروط الاستخدام الخاصة بالمنصة.
  • الالتزام بسياسات KYC/AML عند قبول ودائع من مستخدمين حقيقيين.
  • راجع سياسة exchanges حول cross-account behavior وcopy-trading.
